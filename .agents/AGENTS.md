# Workspace Rules

## 1. UI/UX 및 디자인 철학
- **사용자 편의성 최우선:** 모든 검토는 UI, UX 측면에서 사용자의 편의성을 최우선으로 고려한다.
- **초심자 중심 설계 (Beginner-first):** 항상 가장 초심자의 입장에서 디자인을 구성해야 한다. 
  - 기본 설정만으로도 작업 종료 시까지 만족스럽게 작동하는 필수 요소들로만 구성할 것.
  - O/X를 묻는 불필요한 체크박스 등 초심자가 판단하기 어려운 선택지를 주지 말 것.
  - 숙련자를 위한 기능은 초심자에게 고민의 여지를 주지 않도록 '고급 설정' 등으로 분리하여 선택적으로 적용하게 할 것.
- **시각적 흐름 (Visual Flow):** UI는 시각적으로 단방향(위에서 아래로, 왼쪽에서 오른쪽으로)으로 자연스럽게 구성되어야 한다.
- **웹의 관습 준수:** 모든 UI는 웹의 보편적 관습(예: 입력 시 Enter 확정, ESC 취소)을 최대한 따른다. 만약 일반적이지 않은 UI를 제안할 경우, UX 측면에서 더 나은 대안이라는 명확한 설명과 논리를 제시해야 한다.
- **대안 제시:** 항상 장점과 단점을 분석하고 어떤 방향이 더 일반적이고 자연스러운지 대안을 제시한다.

## 2. 시나리오 기반 심층 검토 (Scenario-based Review)
- **Before/After 검토:** 설계가 변경되었을 때, 사용자가 실제로 사용하게 되는 상황의 '시나리오'를 구체적으로 구성한다.
- **행동 패턴 가정:** 해당 시나리오에서의 사용자 행동 패턴을 논리적으로 가정하고, UI/UX가 이에 어떻게 반응할지를 변경 전(Before)과 변경 후(After)로 나누어 입체적으로 비교 검토한다.
- **리스크 헷징(Risk Hedging):** 이 과정에서 발생할 수 있는 '논리적 오류', '시각적 착각', '의도하지 않은 프로세스 발생 가능성'을 예외 없이 전부 확인하고 대비책을 마련한다.

## 3. 코드 적용 및 리뷰 원칙
- **사이드 이펙트 방지:** 코드를 적용할 때 한 곳을 바꿈으로써 다른 기능이 연쇄적으로 망가지지 않는지 항상 검토하고 방지책을 세운다.
- **무결성 검증:** 코드 작성에서 문법적 실수나 누락이 일어나지 않는지, 의도하지 않은 오류가 없는지 반드시 스스로 검토한다.

## 4. 커밋 및 협의 프로세스
- **선 보고, 후 커밋:** 설계를 변경하거나 코드를 수정할 때에는 위의 모든 검토(시나리오 시뮬레이션, 장단점 분석, 사이드 이펙트 점검)를 먼저 끝낸 뒤, 항상 사용자와 협의를 거치고 승인을 받은 후에 최종 커밋(배포)한다.
